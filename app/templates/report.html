Introduction

George Lucas’ Star Wars has captivated generations with its expansive storylines, epic scores, and awe-inspiring characters. The series itself has impressed upon society a significant cultural impact with references to the film that are deeply embedded within Western popular culture. The events depicted in the seven films are expansive – spanning millennia, with hundreds of different characters, vehicles and species. However, navigating this expansive behemoth of a saga has proven more difficult than one would think and is often bogged down ,in the now non-canon, expanded universe. It is for this reason that our group set out to fix that and provide users an easy to navigate site that allows for one to appreciate the deeply linked nature of all the films and allows one to appreciate the subtleties of continuity one might have missed in these works of art.

In later stages of the we plan to implement a search functionality to further ease the navigation when it comes to specific shared attributes or to find a specific entry even faster. The front-end theme of the website plays homage to the series adopting the colors and monikers of the films. This especially prevalent in our landing page where the iconic approaching logo is accompanied with a fade in of all of our puns to the episodes titles.

Design

Our API design was outlined in apiary. It uses a simple GET request driven system to allow users to query for information about various planets, characters, and films in the Star Wars universe. The design of the API is very straight forward. To obtain a list of the available planets, characters, or films, simply send a request to http://www.thesweawakens.me/api/planets, http://www.thesweawakens.me/api/characters, or http://www.thesweawakens.me/api/films respectively. When visited these links the user is automatically redirected to the first page of the list. Each page contains 6 elements, you can view the next page by changing the “page=x” value in the URL. A JSON is returned a list of objects with the properties of that object. To get information about a specific member of any of these categories, simply append “/{id-number}” to the catagory’s URL. For instance, to obtain the planet with id 2 send a GET request to http://www.thesweawakens.me/api/planets/2. The result is a JSON containing just the properties of that planet.

It is possible to sort or filter you list queries. Adding a “sortUp” or “sortDown” parameter allows you to view the list in a particular order. Adding a “filterBy” and “filter” parameter allows you to apply a filter. The “filterBy” parameter selects the field of the model to filter on and the “filter” field is the values to filter for. For example http://www.thesweawakens.me/api/planets/?page=1&sortUp=climate&filterBy=name&filter=B-G retrieves the second page of the planets list sorted by climate and only including the planets with names start from B to G.

Search


We built the search functionality mostly by ourselves, without using SQLAlchemy's builtin functions. After pulling all the rows from the database, we tokenized our search phrase and searched for each term across all columns of each row. In doing so, we were able to keep track of number of terms matched in a phrase, which would later be used as part of our criteria for sorting the search results. Our primary factor however, was a weight parameter which we determined by checking whether the matched attribute was the name/title of a model, or simply a secondary attribute (anything besides name/title). If the name/title matched, this search result would be given higher weight thereby pushing it to the top of the search results. We distinguished between AND and OR logic by displaying AND matches before OR matches.

We chose to make our search case-insensitive, for ease of use. After much deliberation, we also chose to make our search results render in the form of a grid, much like the rest of our website, rather than a list as Google would do. This was done to maintain UI consistency across our website, and to maximize context for each search result. By using a grid, we were able to incorporate our existing images, so a user could quickly identify a search result at just a quick glance.

All in all, we found that doing manually implementing search ourselves using mostly Python's builtin functions allowed us to make our own tweaks and rank the results just the way we wanted.

Scraping

In order to scrape both the data and the images we employed a programmatic way to pull all the appropriate information. After we established our four models we would work with, we started the process of pulling the data. In order to accomplish this feat we employed the use of the python libraries Requests and Json. The former was used in order to make repeated get requests, in our case to the star wars api, and the latter was used to parse the returned json and pretty-print and save a collated comprehensive data files. From here all we needed to do was import the files into our database.

Before talking about the deployment steps, here are some key notes about using virtualenv and file naming conventions.

Virtualenv

  1. Something that is useful with pretty much any Python project you undertake is to isolate your development environment using a virtualenv. This way, you can install packages to your project’s specific environment and not have to your machine’s global environment. Having isolated dependencies also makes installing dependencies on your deployed VM a lot easier, as shown in a later step.

  2. To create a virtualenv (named venv) and finally deactivate it, type the following on the Command Line:

    $ virtualenv ~/venv

    $ source ~/venv/bin/activate

    $ deactivate (to exit)

  3. While developing the application, make sure to install all your packages via pip inside the virtualenv.

Project structure

  1. Ensure that the flask app is named ‘application.py’, and that the app object used throughout the lifetime of the app is also named ‘application’.

Deploying on AWS Elastic Beanstalk

  1. We used the Elastic Beanstalk Command Line Interface (CLI), which can be installed via pip by typing

    $ pip install awsebcli

  2. Next, we created a new AWS User to essentially be the admin of this deployed app, instead of using a root (personal) account. This way, we could have all our group members login with admin credentials to the AWS console, thereby decoupling it from our personal accounts. To do this on the AWS console:

    • Services > Security, Identity and Compliance > IAM > Users > Add User

    • Type ‘admin’ for username, and check both the access types (secret key will be required in a later step).

    • You will now be asked to create a group, which is not required but recommended. Click Create Group and give the group AdministratorAccess. Save your changes.

    • Add the admin user to the group you just created.

    • Review and Save the user.

  3. No further work is required on the AWS Console, as we can now initialize Elastic Beanstalk via the Command Line. Before doing so, we want to export all our Python dependencies to a requirements.txt file, which the Beanstalk will look up and install any time the app is deployed. So inside the virtualenv, type

    $ pip freeze > requirements.txt

  4. Initialize Beanstalk by typing

    $ eb init

    You should now see a list of location options to choose from, that looks like this:

    Select a default region

    1) us-east-1 : US East (N. Virginia)

    2) us-west-1 : US West (N. California)

    3) us-west-2 : US West (Oregon)

    4) eu-west-1 : EU (Ireland)

    5) eu-central-1 : EU (Frankfurt)

    6) ap-south-1 : Asia Pacific (Mumbai)

    7) ap-southeast-1 : Asia Pacific (Singapore)

    8) ap-southeast-2 : Asia Pacific (Sydney)

    9) ap-northeast-1 : Asia Pacific (Tokyo)

    10) ap-northeast-2 : Asia Pacific (Seoul)

    11) sa-east-1 : South America (Sao Paulo)

    12) cn-north-1 : China (Beijing)

    13) us-east-2 : US East (Ohio)

    14) ca-central-1 : Canada (Central)

    15) eu-west-2 : EU (London)

    We chose option 2, but just choose whatever location is closest to you.

  5. You should now see the following error message “You have not yet set up your credentials or your credentials are incorrect. You must provide your credentials.” Don’t fret, you’re on the right track!

  6. Go back to the AWS Console and find the User that you created earlier in Step 2. Copy and paste your aws-access-id (20-digit id) and aws-secret-key (40-digit key), as prompted by the command line.

  7. Enter the application name on prompt (leave blank for default name).

  8. Type Y, when asked “It appears you are using Python. Is this correct?”

  9. Select your Python platform. We chose Python 3.4 but make sure yours is accurate.

  10. This step is optional, but recommended. You can enable SSH for your AWS EB instance by creating a new key pair (or using an existing one). If you’re not sure, you can always do this on the AWS console at any time.

  11. Almost done! To launch the instance, type

    $ eb create

  12. Chose a DNS CNAME for your application, which is where your website will exist after the first deploy and up until you register a custom domain that maps to this instance. We left it to be the default value, as it was later mapped to a Namecheap provided custom domain, as shown later in this documentation.

  13. Choose the default load balancer.

  14. Elastic Beanstalk should now start loading up the instance, using the configuration that’s been specified in the steps above. This can take up to a few minutes, and up to 10 minutes after that for the server to finish booting up. This wait can be annoying, so perhaps just step away from your machine for about 15 minutes.

  15. Open up your browser, and go to {C-NAME}.elasticbeanstalk.com. Your website should now be live!

  16. Any time you change your files in the future, you can simply type

    17. $ eb deploy

  17. to deploy your changes to your EB instance. It’s always a good idea to test your changes locally before deploying.

  18. If you’re using Git, note that EB will only deploy your last commit and not any uncommitted files. If you’re not using Git, why are you not using Git?

Using a custom domain with Namecheap

  1. Get a free domain from https://nc.me using a student email, or just go to https://www.namecheap.com (paid)

  2. Confirm your contact information and click the link emailed to you.

  3. Login and select your domain on the Namecheap dashboard.

  4. Go to Manage > Advanced DNS.

  5. Create a new CNAME record with host = “www” and value = “{EB CNAME}.elasticbeanstalk.com.”

  6. Create a new URL Redirect record with host = “@” and value = “http://www.{namecheap domain}.me”

  7. Note: Don’t actually put quotations when typing.

  8. Save your changes, and wait for at least 30 minutes for the Namecheap domain to work. According to their documentation, it can take up to 24 hours for the domain to map to your environment.

Thanks to https://colintoh.com/blog/map-custom-domain-to-elastic-beanstalk-application for the Namecheap tutorial!
